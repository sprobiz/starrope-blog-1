웹 API 보안 인증 완전 가이드: JWT·OAuth 2.0·API Key 방식 비교와 실전 적용

작성자: Starrope • 2026. 06. 30 • 읽는 시간: 약 6분
본문 상단 반응형 광고

웹 서비스를 개발할 때 가장 민감하고 중요한 부분이 바로 인증(Authentication)과 인가(Authorization)입니다. "이 요청이 진짜 내 사용자로부터 온 것인가?", "이 사용자는 이 자원에 접근할 권한이 있는가?"를 검증하는 로직이 허술하면 개인정보 유출, 계정 탈취, 서비스 파괴 등 치명적인 보안 사고로 이어집니다. 이번 포스팅에서는 현대 웹 개발에서 가장 널리 사용되는 세 가지 API 인증 방식인 JWT, OAuth 2.0, API Key의 원리와 차이점, 그리고 각 방식의 보안 강화 방법을 체계적으로 정리합니다.

1. JWT(JSON Web Token): 상태 없는 서버를 위한 자가 포함 인증 토큰

JWT(제이더블유티)는 JSON 형식의 데이터를 Base64로 인코딩한 후 서버의 비밀 키로 서명한 문자열입니다. 헤더(Header), 페이로드(Payload), 서명(Signature) 세 부분이 점(.)으로 구분되어 있습니다. REST API에서 세션 기반 인증을 대체하는 가장 인기 있는 방식입니다.

  • 동작 원리: 사용자가 로그인하면 서버는 사용자 정보를 담은 JWT를 생성해 클라이언트에 전달합니다. 이후 모든 API 요청 시 HTTP 헤더에 Authorization: Bearer {토큰}을 포함시키면 서버는 토큰의 서명을 검증하여 사용자를 식별합니다.
  • Stateless의 장점: 서버가 세션 정보를 저장하지 않아도 됩니다. 토큰 자체에 모든 정보가 담겨 있어 수평 확장(Scale-out)이 용이하고, 마이크로서비스 아키텍처에 적합합니다.
  • 보안 주의사항: JWT는 서명만 되어 있을 뿐 암호화되지 않으므로 페이로드에 비밀번호 같은 민감 정보를 담아서는 절대 안 됩니다. Access Token은 만료 시간을 짧게(15분~1시간), Refresh Token은 안전한 HttpOnly 쿠키에 저장하는 것이 보안 Best Practice입니다.
"JWT를 LocalStorage에 저장하는 것은 XSS(Cross-Site Scripting) 공격에 노출되는 지름길입니다. Access Token은 메모리에, Refresh Token은 HttpOnly 쿠키에 저장하세요."

2. OAuth 2.0: 소셜 로그인과 써드파티 API 연동의 표준 프로토콜

OAuth 2.0은 인터넷 사용자의 계정 정보를 서드파티 앱에 직접 노출하지 않고, 구글·카카오·네이버 같은 신뢰할 수 있는 인증 제공자(Identity Provider)를 통해 대신 인증을 처리하는 표준 권한 위임 프레임워크입니다.

  • Authorization Code Flow: 가장 안전한 OAuth 흐름입니다. 사용자가 구글 로그인을 클릭하면, 구글이 인증을 처리하고 단기 Authorization Code를 우리 서버에 전달합니다. 서버는 이 코드를 실제 Access Token으로 교환합니다. 코드가 클라이언트 측에 노출되지 않아 가장 안전합니다.
  • Scope 권한 제어: scope=email profile처럼 요청하는 권한 범위를 명시합니다. 필요한 최소한의 권한만 요청하는 것이 보안 원칙입니다. 사용자는 어떤 정보를 공유할지 명시적으로 동의하게 됩니다.
  • PKCE(Proof Key for Code Exchange): SPA나 모바일 앱처럼 서버 비밀 키를 안전하게 보관할 수 없는 환경에서는 PKCE 확장을 반드시 사용해야 Authorization Code Interception 공격을 예방할 수 있습니다.

3. API Key 인증: 서버 간 통신의 심플하고 강력한 선택

API Key는 특정 클라이언트를 식별하는 고유한 문자열입니다. JWT의 복잡한 서명 검증 없이 단순히 요청에 포함된 키 값이 데이터베이스의 유효한 키와 일치하는지 확인하는 방식으로 작동합니다. B2B SaaS 서비스의 외부 API, 서버 간(M2M) 통신, 개발자 도구 연동에서 가장 많이 사용됩니다.

  • 전달 방식: HTTP 헤더(X-API-Key: your-key), 쿼리 파라미터(?api_key=your-key), 또는 Basic Auth 형태로 전달합니다. 보안상 헤더 방식이 가장 권장됩니다. 쿼리 파라미터는 서버 로그에 키가 노출될 위험이 있습니다.
  • Rate Limiting 필수: API Key 인증 시스템에는 반드시 Rate Limiting(속도 제한)을 구현해야 합니다. 단위 시간당 요청 횟수를 제한하여 무차별 대입 공격(Brute Force)과 DDoS를 방어합니다.
  • 키 순환과 권한 범위: API Key는 주기적으로 갱신(Rotation)하고, 가능하다면 각 키에 접근 가능한 API 엔드포인트와 데이터 범위를 제한하는 세분화된 권한 설계를 적용하는 것이 보안 Best Practice입니다.

마치며

올바른 인증 방식 선택은 서비스의 특성에 따라 달라집니다. 사용자 로그인이 있는 웹/앱 서비스라면 JWT, 소셜 로그인이나 서드파티 연동이 필요하다면 OAuth 2.0, 서버 간 통신이나 개발자 API 제공이 목적이라면 API Key가 일반적으로 최적의 선택입니다. 어떤 방식을 선택하든 HTTPS 필수 적용, 토큰/키의 안전한 저장, 최소 권한 원칙, 로그와 모니터링 구축이라는 보안 4대 원칙은 반드시 지켜야 합니다. 보안은 기능을 만들고 나중에 추가하는 것이 아니라, 설계 단계부터 함께 고려하는 'Security by Design' 철학을 기억하세요.
